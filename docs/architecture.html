Estimat – Planning Poker

En Progressive Web App for sprint-estimering i Gjensidige. Fasilitator oppretter en sesjon og deler en 4-tegns kode – deltakerne stemmer anonymt på størrelse og forretningsverdi, og resultater avsløres i sanntid for hele teamet.

React 19 TypeScript 5.8 Vite 6 Tailwind CSS Supabase Vercel Cloudflare PWA
2 Systemarkitektur

Overordnet arkitektur

Appen er en klientside SPA som kommuniserer direkte med Supabase for data og sanntid. Vercel håndterer hosting og eksponerer én serverless funksjon for bot-beskyttelse. All realtime-flyt går via Supabase Postgres Changes over WebSocket tilbake til nettleseren.

7 Realtime

Supabase Realtime-arkitektur

Appen bruker tre uavhengige Supabase Realtime-kanaler basert på Postgres Changes. Alle klienter abonnerer, og sesjonstilstand propageres øyeblikkelig uten polling.

session-watch:{id}
Lytter på oppdateringer til sesjonen: started, votes_revealed, current_round, status, consensus_streak. Brukt av useSession.
UPDATE
participants:{id}
Lytter på nye deltakere og navneendringer. Brukt av useRealtimeParticipants. Vises i fasilitatorens Dashboard.
INSERT UPDATE
votes:{id}:{round}
Lytter på nye stemmer for aktuell runde. Tilkoblet per sesjon+runde, resettes ved ny runde. Brukt av useRealtimeVotes.
INSERT
✅

Subscribe-then-fetch-mønster: Alle realtime-hooks abonnerer først og henter initial data etter at subscription er bekreftet (status === 'SUBSCRIBED'). Dette eliminerer race conditions der hendelser kan gå tapt mellom fetch og subscription-oppstart.

ℹ️

Ingen fallback-polling. Med REPLICA IDENTITY FULL på plass er polling fjernet – det sparte batteri og nettverkstrafikk unødvendig. Duplikat-guard i INSERT-handlers (id-sjekk) håndterer sjeldne race conditions ved re-subscribe.

8 Sikkerhet

Sikkerhetsarkitektur

🤖

Cloudflare Turnstile

Alle brukere må bestå Turnstile-sjekk ved landing. Token verifiseres server-side via Vercel serverless function – secret key er aldri eksponert til nettleseren (ingen VITE_-prefix).

Rate limiting: maks 10 forsøk/minutt per IP (in-memory per function instance). Graceful degradation er fjernet bevisst – nettverksfeil mot Cloudflare blokkerer videre tilgang.

🔒

Supabase RLS – operasjonsspesifikke policyer

Migrasjon 005 erstattet åpne ALL-policyer med operasjonsbegrensede regler: ingen kan DELETE sesjoner/deltakere, og stemmer er INSERT-only (kan ikke endres etter avgivelse).

Uten auth kan ikke RLS knyttes til individuelle brukere – person-spesifikk kontroll håndteres av app-logikk via participant-ID lagret i sessionStorage.

🆔

SessionStorage-identitet

Deltaker-ID og sesjon-ID lagres i sessionStorage (ikke localStorage). Data lever kun i fanen – ingen persistens mellom nettleserøkter. Gjenoppretting ved refresh skjer via DB-oppslag.

Bevisst valg for intern MVP: ingen auth/login reduserer friksjon betraktelig for korte planleggingsseanser.

⚠️

Ingen autentisering – bevisst valg

Appen er designet for internt bruk i Gjensidige uten inlogging. Enkelheten prioriteres over streng tilgangskontroll – en deltaker med riktig join_code kan delta.

join_code er 4 tegn fra 30 mulige tegn → 810.000 kombinasjoner. Ikke kryptografisk sterk, men tilstrekkelig for kortlevde interne sesjoner.

9 PWA

Progressive Web App-funksjonalitet

Appen er en fullverdig PWA – installerbar på mobil, fungerer offline for cachet innhold, og holder skjermen aktiv under avstemning.

⚙️

Service Worker (Workbox)

registerType: 'autoUpdate' med skipWaiting + clientsClaim. Ny Service Worker aktiveres umiddelbart etter deploy. Alle statiske assets pre-cachet; JS-chunks håndteres med NetworkFirst (1 dag TTL) for å alltid laste ferskeste kode.

📋

App Manifest

Navn: Estimat – Planning Poker. Display: standalone. Orientering: portrait. Theme color: #0B1D3A (navy). Ikoner: 192×192, 512×512 og maskable 512×512. Generert med sharp fra SVG-kilde.

💡

Screen Wake Lock

Aktivert på Vote-siden (alltid) og Dashboard-siden (kun aktivt sesjon-view). Re-acquirer automatisk ved tab-bytte via visibilitychange. Feiler stille – ingen krasj ved manglende API-støtte.

🔄

Chunk-feil-håndtering

lazyWithRetry: ved chunk-lastfeil → sjekker sessionStorage, reloader én gang, og kaster videre ved gjentatte feil. ChunkErrorBoundary (class component) fanger feil som slipper gjennom og tvinger reload. Forhindrer hvit skjerm etter deploy.
